## 一、产品概述

### 1.1 核心理念

**最大化每一位剧本杀玩家的个人游戏体验。**

剧本杀的核心价值在于游戏体验本身——推理的快感、情感的共鸣、沉浸的代入感。然而在传统多人场景中，这种体验往往受到严重稀释：

- **水平差异**：玩家推理能力参差不齐，强玩家主导全场，弱玩家沦为旁观者
- **偏好冲突**：有人想深度烧脑，有人只想轻松欢乐，强行凑局导致双方都不满意
- **同频困难**：对剧情的投入度、对角色的代入感、对游戏节奏的期望各不相同
- **组局约束**：时间、地点、人数的协调成本让高质量组局越来越难

单人 AI 剧本杀从根本上消解了这些问题——**玩家不再需要迁就他人，所有的剧本选择、难度设定、游戏节奏，都围绕一个人的体验偏好而精准构建。** 这不是"将就"的独行，而是将个人游戏体验拉升到传统多人模式难以触达的高度。

### 1.2 产品定位

- **目标用户**：所有追求高质量剧本杀体验的玩家——无论是苦于难以组局，还是厌倦了将就型游戏体验，抑或是想在正式组局前深度预习剧本
- **核心价值**：以 AI 多角色协作为手段，将游戏体验的主导权完全交还给玩家个人，实现体验的精准匹配与最大化
- **差异化定位**：不是"独自玩剧本杀"，而是"为你一个人定制的剧本杀"
- **V1 边界**：单人游戏（玩家 + AI 角色群）；多人/CP 本后续迭代

### 1.3 当前实现校准（v1.1）

截至 2026-06-10，当前项目已经从早期“规划型 PRD”进入本地可运行原型阶段。后续需求应以以下已实现能力作为基线，而不是以旧版远期规划作为唯一参照：

- **运行形态**：本地优先的 Next.js 应用，默认使用 Prisma + SQLite，可在无外部模型密钥时通过 mock 模式完整跑通。
- **模型后端**：优先 Step，其次 Anthropic，最后 mock。Step 已用于角色/DM 对话、TTS、ASR 等能力；后续不应在代码里硬编码具体 Step 模型，而应通过配置切换。
- **剧本来源**：已支持内置剧本库、剧本上传、AI 生成剧本入口。内置剧本可作为无配置体验与验收基准。
- **玩家模式**：已支持角色扮演模式与侦探模式，角色扮演模式可进行个人选角。
- **游戏流程**：已具备阶段推进、顺序自我介绍、自由讨论、私聊、线索发布、投票、复盘等基本流程。
- **DM 展示**：DM 长内容应进入右侧主持台；公屏只沉淀摘要，避免 DM 信息刷屏。
- **Agent 轮次**：已引入回合完整性与对话导演雏形，避免角色发言被截断后直接进入下一位，并支持点名回应。
- **语音能力**：已接入 Step TTS/ASR，TTS 队列以“播放时展示消息”为核心体验，避免一次性刷出大量文本。
- **线索卡当前缺口（v1.4.1）**：线索卡已能发布、沉淀到右侧牌堆，并可由玩家以普通文本“展示线索”发送到公屏；但 `ClueCard` 仍缺少媒体字段、视觉批次、展示者/目标/可见范围等事件语义，Agent 只能从普通聊天文本里被动读到线索，不能稳定围绕线索卡讨论，也不能互相展示线索。
- **阶段收敛当前缺口（v1.5.2）**：阶段推进已能展示 DM/Agent 状态和完成度，但判断逻辑主要依赖消息数量、AI 回应数量和线索释放数量；缺少“本阶段议题、已举证线索、当前共识、仍未达成共识的分歧点、玩家提交的阶段结论”等桌游必要结构，导致体验容易变成 DM 说完后大家轮流聊天。
- **线上数据与认证基线（v1.7）**：Vercel 生产环境使用 Supabase Postgres 作为远端数据库；邮箱注册/登录以 Supabase Auth 为身份源，业务库 `User.id` 与 Supabase Auth user id 对齐。未登录仍保留游客模式，便于无账号试玩；登录后剧本、游戏、消息、投票等数据按用户归属隔离。Supabase RLS 用于保护通过 Supabase Data API 访问的业务表，后端 Route Handlers 继续作为游戏编排入口。

后续 PRD 迭代必须围绕“更像剧本杀桌面，而不是聊天页面”推进。

## 二、用户旅程总览

> 旅程从"你今天想要什么体验"出发，而不是从"选剧本"出发。

```
进入游戏
    │
    ├── ① 体验意图确认（首要入口）
    │   系统引导玩家回答：
    │   "你今天想要什么感觉？"
    │   → 烧脑推理 / 情感共鸣 / 轻松欢乐 / 惊悚刺激 / 还原真相
    │   系统根据回答推荐剧本类型 + 难度 + 角色模式（可覆盖）
    │
    ├── ② 选择玩家模式
    │   ├── 角色扮演模式（推荐：想深度代入、有秘密要守护）
    │   └── 侦探模式（推荐：想自由审讯、专注找出真相）
    │
    ├── ③ 选择剧本来源
    │   ├── A. AI 生成剧本（引导问卷，完全定制）
    │   ├── B. 上传剧本（PDF / 文本，已有剧本）
    │   └── C. 内置剧本库（V2）
    │
    ├── ④ 剧本配置确认（预览公共故事 + 确认角色/难度）
    │
    ├── ⑤ 剧本分配（DM 分配角色给各 AI Agent，玩家获得自己的剧本）
    │
    ├── ⑥ 游戏进行（DM 按阶段推进，全程体验自适应）
    │
    ├── ⑦ 投票/结局判定 → DM 汇总结果
    │
    ├── ⑧ 复盘揭秘 → 查看全角色剧本 + 私聊记录 + 胜负分析
    │
    └── ⑨ 体验反馈 → 系统记录偏好，影响下次推荐
```

## 三、剧本来源模块

### 3.1 上传剧本

**支持格式**：PDF、TXT、Markdown、Word（.docx）

**解析目标**：系统需识别并提取以下内容：
- 公共剧本（所有玩家可见的背景故事）
- 各角色私密剧本（含角色背景、秘密、隐藏目标、胜利条件）
- 线索卡列表（物品线索、地点线索、时间线）
- 游戏阶段结构（若剧本中有标注）
- 特殊机制说明（道具卡/事件卡/技能等，若有）

**解析策略**：
- 优先识别结构化标题（如"公共故事"、"角色：XXX"、"胜利条件"等关键词）
- 无法自动识别时，引导用户手动标注关键段落
- 解析结果展示给用户确认，支持人工修正

**剧本校验**（按剧本类型差异化）：
- 检测角色数量是否在支持范围内（3~8人）
- 推理本/硬核本/还原本：必须存在凶手角色，否则提示补充
- 情感本/欢乐本：无需凶手，但必须有核心冲突事件
- 恐怖本：必须有威胁来源（可以是人/超自然/环境），不强制为"凶手"
- 所有类型：检测每个角色是否包含胜利条件（若无，提示用户补充或 AI 自动生成兜底条件）

### 3.2 AI 生成剧本

#### 3.2.1 引导问卷设计（分步交互）

用户通过对话式问卷逐步确认定制维度，系统在每步给出推荐默认值，用户可接受或修改。

**第零步：体验意图（前置引导）**
| 问题 | 选项/输入 | 说明 |
|------|-----------|------|
| 你今天想要什么体验？ | 烧脑推理（享受解谜过程）/ 情感共鸣（沉浸故事体验）/ 轻松欢乐（娱乐放松）/ 惊悚刺激（肾上腺素）/ 还原真相（重建历史） | 决定推荐的剧本类型和后续默认值 |
| 你的剧本杀经验如何？ | 新手（第一次玩）/ 普通玩家（玩过几次）/ 老手（经常玩）/ 硬核玩家（精通推理） | 决定推荐难度和DM引导力度 |
| 你今天有多少时间想投入？ | 轻体验（~1h，随时可结束）/ 标准局（~2h）/ 深度沉浸（3h+，不想被打断）| 影响剧本信息量与复杂度 |

**第一步：基础框架**
| 问题 | 选项/输入 | 说明 |
|------|-----------|------|
| 选择剧本类型 | 推理本 / 硬核本 / 情感本 / 欢乐本 / 恐怖本 / 还原本 | 决定阶段结构 |
| 选择时代背景 | 古代 / 民国 / 现代 / 近未来 / 架空奇幻 / 赛博朋克 / 末世 / 自定义 | |
| 选择地点场景 | 密室 / 庄园 / 邮轮 / 学校 / 医院 / 古镇 / 太空站 / 荒岛 / 自定义 | |
| 角色数量 | 3 / 4 / 5 / 6 / 7 / 8 人 | 含玩家自己 |
| 游戏时长 | 短（~1h）/ 中（~2h）/ 长（~3h+）| 影响剧本信息量 |

**第二步：叙事与风格**
| 问题 | 选项/输入 |
|------|-----------|
| 难度 | 新手友好 / 进阶 / 硬核烧脑 |
| 线索密度 | 稀疏（强推理）/ 适中 / 密集（信息量大）|
| 叙事结构 | 线性单线 / 多线并行 / 回忆录式 |
| 文字风格 | 现代白话 / 文言古风 / 欧美侦探风 / 网文爽感 |
| 情感基调 | 轻松欢乐 / 严肃烧脑 / 催泪悲情 / 惊悚恐怖 / 浪漫唯美 |

**第三步：主题与内容**
| 问题 | 选项/输入 |
|------|-----------|
| 主题类型 | 家庭伦理 / 商业阴谋 / 爱恨情仇 / 校园青春 / 职场争斗 / 历史改编 / 宗教神秘 / 政治权谋 |
| 特殊元素（可多选）| 超自然 / 失忆 / 双重身份 / 连环案 / 穿越 / 无 |
| 翻转设计 | 无翻转 / 单次大翻转 / 多次小翻转 |
| 结局类型 | 唯一真相 / 多结局分支 / 开放式 |

**第四步：角色定制**
| 问题 | 选项/输入 |
|------|-----------|
| 玩家角色身份 | 随机分配 / 我是凶手 / 我是受害者家属 / 我是侦探/目击者 |
| 角色关系复杂度 | 简单清晰 / 网状复杂 |
| 特定角色要求 | 自由填写（如"有一个角色是我的前任"、"有一对姐妹"）|

**第五步：机制与边界**
| 问题 | 选项/输入 |
|------|-----------|
| 特殊机制 | 无 / 道具卡 / 事件卡 / 角色技能 / 状态变化 |
| 内容边界 | 无限制 / 无暴力描写 / 无恐怖元素 / 家庭友好 |
| 个性化参考 | 是否基于历史游戏偏好推荐？/ 是否基于真实历史事件改编？/ 特定 IP 世界观（用户填写）|

#### 3.2.2 剧本生成规范

- 生成内容必须包含：公共故事、每角色私密剧本（含背景/秘密/隐藏目标/胜利条件）、线索卡、阶段结构建议
- 凶手身份在剧本生成时确定，写入 DM 专属知识，不泄露给其他角色 Agent
- 生成后展示公共故事供玩家预览；私密剧本仅在分配后展示给对应角色/玩家

## 四、玩家模式

> 模式选择应服务于玩家的体验目标，而非强制分类。系统在用户旅程第①步完成后，会根据体验意图给出推荐模式，玩家可自由覆盖。

### 4.1 角色扮演模式

**适合人群**：想要深度代入角色、体验身份扮演张力、有秘密需要守护的玩家。

- 玩家持有一个具体角色的完整私密剧本
- 玩家有自己的秘密需要隐瞒，有隐藏目标需要达成，有独立胜利条件
- 玩家参与投票，也接受其他角色的质疑和审讯
- 胜负判定基于角色胜利条件是否达成
- AI 角色将视玩家为"同场竞技的角色"，而非旁观者，会对玩家产生怀疑、结盟、试探等行为

**典型胜利条件示例**：
- 凶手角色：游戏结束时未被多数票投出
- 受害者家属：找出真凶并成功投票指出
- 知情者：在不暴露核心秘密的前提下完成特定任务
- 情感本角色：完成情感目标（如与某角色和解、说出心中真相）
- 普通角色：完成个人隐藏任务（如拿回某物、获得某人信任）

### 4.2 侦探模式

**适合人群**：想专注于推理审讯、不想受角色身份约束、以"破案"为核心乐趣的玩家。

- 玩家没有角色身份，以外来侦探身份进入游戏
- 玩家可自由与任何 AI 角色进行公聊或私聊审讯，不受发言顺序约束
- AI 角色知晓"有一位外来侦探"，对玩家的配合程度由各自角色设定决定（有人配合、有人抵触）
- 玩家不参与 AI 角色之间的私密互动，但游戏结束后复盘可查

**侦探模式胜利条件按剧本类型差异化**：

| 剧本类型 | 侦探模式胜利条件 |
|---------|---------------|
| 推理本 / 硬核本 | 在投票阶段正确指出凶手 |
| 还原本 | 提交的时间线还原准确度达到80%以上（DM 判定） |
| 情感本 | 成功揭露核心情感真相，并让至少一位角色做出真实表态 |
| 欢乐本 | 无严格胜负，以"最欢乐的侦探记录"为结局（DM 颁发趣味称号） |
| 恐怖本 | 找到逃脱路径或揭露威胁来源，并成功"存活"至结局 |

## 五、游戏类型与阶段结构

> DM Agent 严格按阶段推进，每阶段结束时由 DM 宣布进入下一阶段。

### 5.1 推理本

**核心特征**：逻辑推理为主，线索严谨，存在唯一真相。

```
① 阅本阶段
   - 玩家阅读角色剧本（角色模式）或公共故事（侦探模式）
   - DM 宣布游戏规则与流程

② 入戏自我介绍阶段
   - 每个角色（含玩家）依次进行角色介绍（可隐瞒秘密）

③ 自由交流阶段（1~2轮）
   - 公开讨论：所有角色在公共频道发言
   - 私聊开放：玩家可主动与任意角色私聊

④ 独立搜证阶段
   - DM 发布场景线索卡（分批解锁）
   - 玩家可对线索提出质疑或结合已有信息推理
   - AI 角色对线索作出反应（可能说谎或隐瞒）

⑤ 公开质询阶段
   - 任何角色可对其他角色提出质疑
   - 被质询角色必须回应（可选择隐瞒）

⑥ 最终陈词阶段
   - 每个角色发表最终陈词

⑦ 投票指凶阶段
   - 玩家提交投票；AI 角色模拟投票
   - DM 汇总并公布结果

⑧ 复盘揭秘阶段
   - 揭示全部真相、各角色完整剧本
   - 展示私聊记录
   - 宣布各角色胜负结果
```

### 5.2 硬核本

**核心特征**：信息量极大，多条证据链，需严密逻辑推导，设置中间推理验证节点。

```
① 阅本阶段（时间更充裕）
② 入戏自我介绍阶段
③ 第一轮自由交流 + 搜证
④ 【中间推理节点 A】
   - DM 发起"阶段性推理提交"：玩家提交当前对案情的判断
   - DM 给出"方向正确/需重新考虑"的提示（不透露真相）
⑤ 第二轮深度搜证
   - 解锁更深层线索（需前置线索作为触发条件）
⑥ 【中间推理节点 B】
   - 再次提交推理，DM 给出进一步提示
⑦ 公开质询阶段（多轮）
⑧ 最终陈词
⑨ 投票指凶
⑩ 复盘揭秘
```

### 5.3 情感本（哭哭本）

**核心特征**：剧情体验优先，弱化推理，强化角色间的情感互动与戏剧冲突。

```
① 阅本阶段
② 情感入戏阶段
   - 角色介绍侧重于情感背景，而非案情信息

③ 自由交流阶段（情感引导）
   - DM 适时推出"情感触发事件"（如发现旧照片、收到信件）
   - AI 角色对情感事件作出情绪化反应

④ 深度对话阶段
   - 玩家可与角色进行深度一对一私聊
   - AI 角色可能在深度对话中透露更多秘密

⑤ 情感高潮事件
   - DM 触发核心戏剧冲突（可能是关键真相揭示、情感决裂等）

⑥ 各角色表态阶段
   - 每个角色表达立场（不强制为指凶，可为情感选择）

⑦ 投票/情感判定阶段
   - 投票权重弱化；侧重于"你是否达成了情感目标"

⑧ 复盘揭秘（含情感线索梳理）
```

### 5.4 欢乐本

**核心特征**：轻松娱乐，随机事件制造混乱，不强调逻辑严谨。

```
① 快速阅本阶段（时间压缩）
② 搞笑入戏阶段（角色夸张自我介绍）
③ 自由混乱交流阶段
   - 随机事件卡触发（由 DM 随机发布，如"所有人必须用第三人称说话10分钟"）

④ 线索收集（轻量版，线索带娱乐性）

⑤ 随机事件高潮
   - DM 触发1~2个随机剧情转折事件

⑥ 欢乐指凶阶段
   - 投票前每人说一句最搞笑的指控理由

⑦ 投票指凶

⑧ 揭秘（含游戏中精彩发言回顾）
```

### 5.5 恐怖本

**核心特征**：事件驱动，DM 随时触发恐怖事件，设置生存/逃脱判定节点。

```
① 阅本阶段（强调氛围渲染）
② 入戏阶段（DM 用环境描写营造氛围）
③ 自由探索阶段
   - 玩家/角色在不同场景探索
   - DM 根据探索行为触发恐怖事件（随机+剧情触发）

④ 【生存判定节点 A】
   - DM 评估角色状态（是否触发危险事件）
   - 可能触发角色"濒危"状态，影响其后续行动

⑤ 继续探索 + 真相追踪

⑥ 【逃脱/对抗判定节点】
   - 根据剧本类型：找出元凶并对抗 / 找到逃脱路径
   - 玩家需做出关键抉择（影响结局分支）

⑦ 结局揭示（多结局分支）
   - 生存结局 / 牺牲结局 / 真相揭露结局

⑧ 复盘（含恐怖事件触发逻辑揭示）
```

### 5.6 还原本

**核心特征**：核心目标是还原历史真相，侧重时间线重建与证人陈述。

```
① 阅本阶段
② 入戏阶段（强调各角色与事件的关联）

③ 时间线梳理阶段
   - DM 发布事件碎片（时间节点乱序）
   - 玩家需整理时间线顺序

④ 证人陈述轮（多轮）
   - 每个角色依次陈述自己在特定时间点的行动
   - 其他角色和玩家可提出质疑

⑤ 对质与核实阶段
   - 发现陈述矛盾时，DM 组织对质
   - 可调取"物证"（DM 发布对应线索卡）

⑥ 时间线重建提交
   - 玩家提交完整时间线还原

⑦ 真相确认 + 指认阶段

⑧ 复盘（展示完整时间线真相与各角色陈述对比）
```

## 六、Agent 设计规范

### 6.1 角色 Agent

**数量**：每个非玩家角色对应一个独立 Agent（3~8人本中，玩家占1位，其余为 AI Agent）

**信息隔离原则**：
- 每个角色 Agent 的上下文仅包含：公共剧本 + 自己的私密角色剧本
- 严禁加载其他角色的私密信息
- 不知晓其他角色的真实身份和秘密

**Prompt 注入结构**（每个角色 Agent 独立）：
```
[系统角色定义]
你是一个剧本杀游戏中的角色扮演 AI。
你的任务是完全代入角色，模拟这个角色在游戏中的言行举止。

[公共剧本]
{公共背景故事}

[你的角色信息]
姓名：{角色名}
性别：{性别}
职业：{职业}
外显性格：{公开性格描述}
背景故事：{角色背景}
你知道的秘密：{私密信息}
你不知道的事：{该角色在剧本中的信息盲区}
隐藏目标：{需要在游戏中达成的目标}
胜利条件：{判定胜负的标准}

[行为约束]
1. 你必须保护自己的秘密，不能主动透露
2. 可以说谎，但说谎要符合角色逻辑
3. 你不知道其他角色的剧本内容，根据游戏中的对话推断
4. 当被追问时，可以选择回避、转移话题、或部分透露
5. 情绪和反应需符合角色性格设定

[当前游戏阶段]
{DM 实时注入的阶段信息}
```

**角色 Agent 行为能力**：
- 参与公共频道讨论
- 接受并发起私聊
- 对线索作出反应
- 参与质询（质问他人/被质问）
- 提交模拟投票
- 发表最终陈词

### 6.2 DM Agent

**定位**：游戏上帝，全局协调者，不扮演任何剧本角色。

**信息权限**：
- 持有所有角色的完整剧本（含秘密、胜利条件）
- 持有完整案情真相（凶手、动机、作案过程）
- 持有游戏阶段结构与推进规则
- 实时追踪游戏状态（当前阶段、已公开线索、各角色发言摘要）

**DM Prompt 结构**：
```
[DM 系统定义]
你是剧本杀游戏的 DM（主持人/游戏主控），拥有全部真相。
你的职责是控制游戏节奏、维护游戏秩序、确保体验质量。

[完整剧本信息]
{所有角色剧本 + 完整案情真相}

[游戏阶段结构]
{当前剧本类型对应的阶段流程}

[DM 职责]
1. 阶段推进：按时/按条件宣布进入下一阶段
2. 线索发布：在适当时机发布线索卡
3. 冲突调解：防止讨论陷入僵局，适时引导
4. 特殊机制触发：触发事件卡/恐怖事件/随机事件等
5. 投票汇总：收集玩家投票 + 各 AI 角色模拟投票，计算结果
6. 反作弊：确保 AI 角色不意外泄露不该知道的信息
7. 体验把控：当玩家明显卡住时，给出适度引导提示（不直接透露答案）

[当前游戏状态]
{实时游戏状态快照}
```

**DM 职责补充——玩家体验自适应（核心能力）**：

DM 需要持续评估玩家的体验状态，并动态响应，这是"最大化体验"的关键执行者：

| 体验信号 | DM 识别方式 | DM 响应策略 |
|---------|-----------|-----------|
| 玩家卡顿/挫败 | 连续N轮问题无实质推进，或玩家使用"不知道"/"没思路"等表达 | 给出引导性问题（不透露答案），或主动发布额外线索 |
| 玩家无聊/推进过快 | 玩家快速消耗所有线索，或频繁催促进入下一阶段 | 触发隐藏支线事件，增加复杂度，或提前释放更深层线索 |
| 玩家高度投入 | 玩家发出大量推理性发言，主动私聊多个角色 | 维持现状，减少干预，让玩家主导节奏 |
| 玩家要求暂停 | 玩家发送暂停指令 | 保存游戏状态，AI 角色进入"待机"状态 |

**DM 发言风格自适应（按剧本类型）**：
- 推理本/硬核本：严肃、精准、克制，不渲染情绪
- 情感本：温柔、有共情感，描述氛围时带情绪色彩
- 欢乐本：活泼、带梗、适度幽默
- 恐怖本：营造恐惧氛围，描述细节，制造紧迫感
- 还原本：像记者/法官，强调证据和逻辑

**DM 触发条件（规则引擎）**：
- 时间触发：阶段超时自动推进
- 条件触发：玩家完成特定动作（如"提交推理"、"搜查完毕"）
- 卡顿触发：连续N轮无实质进展时，DM 介入引导
- 事件触发：恐怖本/欢乐本中的随机/剧情事件
- 体验触发：检测到体验信号时的自适应响应

### 6.3 Agent 间通信架构

```
玩家
 ├── 公共频道 ←→ 所有角色 Agent（全部可见）
 └── 私聊频道 ←→ 指定角色 Agent（一对一，其他角色不可见）

角色 Agent A ←→ 私聊频道 ←→ 角色 Agent B
（由 DM 协调触发，玩家不可见，但游戏结束后复盘可查）

DM Agent
 ├── 监听所有频道（全局可见）
 ├── 公共广播（向所有人发布阶段通知、线索等）
 └── 私信玩家（给予引导提示）
```

## 七、结局判定机制

> 结局判定不统一为"投票指凶"，而是根据剧本类型和玩家模式差异化处理，确保每种体验都有匹配的收尾方式。

### 7.1 结局判定类型

| 剧本类型 | 结局判定方式 |
|---------|------------|
| 推理本 / 硬核本 | 标准投票制：玩家 + AI 角色各投1票，DM 汇总，得票最多者即为"众矢之的" |
| 还原本 | 时间线提交制：玩家提交完整时间线，DM 对照真相评分 |
| 情感本 | 情感目标达成制：每个角色判定情感目标是否达成，无强制投票（可选举行象征性的"真相表决"） |
| 欢乐本 | 轻量投票 + 趣味判定：投票保留但附带"搞笑指控理由"，DM 额外颁发趣味称号 |
| 恐怖本 | 存活/逃脱判定：DM 根据玩家行为路径判定存活状态，揭示结局分支 |

### 7.2 标准投票规则（推理本/硬核本）

DM 宣布进入投票阶段后：
1. 每个 AI 角色 Agent 根据自身获取的信息，独立提交投票（附带简短理由）
2. 玩家提交投票
3. DM 汇总所有投票
4. 得票最多者被判定为"众矢之的"
5. 若玩家处于角色扮演模式，玩家投票计为1票（与 AI 同权）
6. 平票时：DM 组织一轮加时质询，或按剧本预设规则处理

### 7.3 胜负判定

DM 在揭秘阶段对每个角色（含玩家）逐一宣布胜负结果，依据各自的胜利条件：
- 是否完成隐藏任务
- 是否成功隐瞒/揭露秘密
- 是否在投票/判定中达成目标
- 情感本：是否达成情感目标（DM 综合全场对话判断）

## 八、对话与交互规范

### 8.1 频道类型

| 频道 | 参与者 | 可见范围 |
|------|--------|---------|
| 公共大厅 | 所有角色 + 玩家 + DM（广播） | 全员可见 |
| 私聊频道 | 玩家↔某角色 / 角色↔角色 | 仅双方可见；游戏结束后复盘全部公开 |
| DM 提示频道 | DM → 玩家 | 仅玩家可见（相当于 DM 附耳提示） |

### 8.2 私聊触发规则

- **玩家发起**：玩家可在任意阶段主动向某角色发起私聊（部分剧本类型可能有限制）
- **角色发起**：AI 角色在以下情况可主动向玩家或其他角色发起私聊：
  - 剧本中有指定的"秘密联络"设定
  - 游戏进入特定阶段（如情感本的深度对话阶段）
  - AI 角色判断需要私下联盟或透露信息
- **角色间私聊**：由 DM 在适当时机协调触发（玩家不可见，但知晓"有私聊发生"的提示）

### 8.3 交互节奏管理

**DM 侧控制**：
- DM 负责控制每个阶段的发言顺序（可设置自由发言/轮流发言两种模式）
- 若某个 AI 角色长时间无响应，DM 自动代为给出简短回应
- 若玩家长时间无操作，DM 发送轻提示

**玩家侧节奏控制权**（体验最大化的关键）：

玩家可在游戏中随时发出以下指令，DM 必须响应：

| 玩家指令 | DM 响应 |
|---------|--------|
| "我需要一点提示" | DM 给出1条引导性问题，不直接透露答案 |
| "我想暂停游戏" | 保存完整游戏状态，所有 AI 角色进入待机；下次进入时原样恢复 |
| "我觉得太难了，能降低一点难度吗" | DM 调整后续线索释放节奏（加快），增加一条额外提示线索 |
| "我想跳过这个阶段" | DM 评估是否可以跳过（不影响剧情完整性则允许，否则给出理由） |
| "我想多了解一下[某角色]" | DM 安排该角色主动找玩家发起对话 |
| "我有点出戏，提醒我一下背景" | DM 以旁白形式简短回顾当前剧情状态 |

## 九、游戏记录与复盘系统

### 9.1 游戏记录内容

每局游戏自动保存：
- 游戏基础信息（剧本名/类型/角色/日期/时长）
- 完整对话记录（公共频道 + 所有私聊频道，含时间戳）
- 线索发布时间线
- 投票记录（各角色投票内容及理由）
- 胜负结果

### 9.2 复盘功能

游戏结束后，玩家可进入复盘页面：
- **真相揭示**：完整案情真相展示
- **全角色剧本**：所有角色的私密剧本全部公开
- **私聊记录回放**：按时间线查看所有私聊内容（包括角色间私聊）
- **关键节点标注**：DM 标注的关键转折点
- **胜负分析**：每个角色胜利条件的达成情况分析
- **AI 行为点评**：DM Agent 对游戏中精彩/关键发言的点评

### 9.3 游戏后体验反馈

游戏结束后，系统邀请玩家进行简短反馈（可跳过），反馈数据用于优化下次体验推荐：

| 反馈维度 | 形式 |
|---------|------|
| 整体体验评分 | 1~5星 |
| 最喜欢的环节 | 多选（推理过程/角色互动/剧情反转/沉浸氛围） |
| 难度感受 | 太简单 / 刚好 / 有点难 / 太难 |
| 是否想再玩同类剧本 | 是 / 否 |
| 开放留言 | 可选填写 |

**系统根据反馈生成下次推荐**：
- 结合本次反馈 + 历史游戏记录，在下次进入游戏时自动推荐体验意图和剧本类型
- 若玩家多次选择同类体验，系统提示"你可能也会喜欢 [相关类型]" 扩展体验边界

### 9.4 历史游戏列表

- 按时间倒序展示历史游戏
- 可按剧本类型/胜负结果/体验评分筛选
- 支持再次进入复盘
- 显示每局的"体验标签"（如：经典翻转、催泪结局、成功破案）

## 十、用户账户系统

### 10.1 账户功能（V1 基础版）

- 注册/登录（邮箱 + 密码 / 三方 OAuth）
- 个人游戏记录存储
- 历史剧本管理（上传的/AI生成的）
- **体验偏好档案**（由系统根据游戏记录和反馈自动构建，玩家也可手动调整）：

| 偏好维度 | 说明 |
|---------|------|

### 10.2 数据隔离

- 每个用户只能访问自己的游戏记录和剧本

## 十一、不支持场景（V1 明确排除）

| 场景 | 原因 | 规划 |
|------|------|------|
| 机制本/阵营本 | 依赖多真人社交互动，AI 无法还原社交博弈 | V2 多人模式 |
| 情侣 CP 本 | 需双人参与 | V2 双人模式 |
| 多人实时同局 | 当前架构聚焦单人 + AI 多 Agent | V2 多人模式 |
| 复杂实时语音对话 | 当前已支持 TTS/ASR，但 realtime 低延迟语音尚未完成 | V1.x 接入 Step realtime |

## 十二、非功能性需求

| 指标 | 要求 |
|------|------|
| 响应延迟 | Agent 回复 ≤ 8 秒（正常情况） |
| 并发 | V1 支持单用户单局（无需多人并发） |
| 数据持久化 | 游戏记录永久保存 |
| 内容安全 | 生成内容过滤敏感违规信息 |
| 隐私 | 用户剧本和游戏记录加密存储，不用于模型训练 |

## 十三、体验质量保障机制

> 产品核心目标是"最大化体验"，需要明确体验质量的守护机制，而不仅靠功能堆砌。

### 13.1 角色一致性保障

AI 角色的"出戏"是最严重的体验破坏因素：
- 每个角色 Agent 在整局游戏中必须保持性格、立场、秘密持有状态的一致性
- DM 负责监控角色行为，若检测到明显前后矛盾（如已说谎的事实被推翻），DM 介入提示该角色"收回"或"补充解释"
- 角色 Agent 不得以 AI 身份回应，不得在任何情况下打破第四面墙

### 13.2 沉浸感保护

- 系统不在游戏进行中弹出任何与游戏无关的 UI 通知
- DM 提示统一通过游戏内频道传达，不使用系统弹窗
- 游戏界面设计上屏蔽系统元素干扰（全屏/沉浸模式优先）

### 13.3 体验兜底机制

若出现以下情况，系统有明确的兜底行为：

| 异常情况 | 兜底行为 |
|---------|---------|
| AI 角色连续回复质量低下（过短/无意义） | DM 重新触发该角色，注入更具体的行为指令 |
| 剧本生成失败或内容不完整 | 提示玩家重新生成，保留已确认的问卷答案 |
| 玩家长时间无法推进且多次求助 | DM 提供"剧情加速通道"：直接给出当前阶段关键线索 |
| 网络中断导致游戏中断 | 游戏状态按最后一条消息保存，重连后原样恢复 |

### 13.4 体验质量指标（内部监控）

| 指标 | 目标 |
|------|------|
| 角色回复响应时间 | ≤ 8秒（P90） |
| 角色前后矛盾发生率 | < 5% 的游戏局次 |
| 玩家主动暂停率 | 监控趋势（异常升高说明体验有问题） |
| 游戏完成率（未中途放弃） | > 80% |
| 复盘完成率 | > 60%（看完复盘 = 整体体验满意） |
| 体验评分均值 | ≥ 4.0 / 5.0 |

## 十五、v1.2 桌面化游戏体验与自动控场需求

### 15.1 体验目标

当前迭代的目标是把产品从“聊天页面”升级为“剧本杀桌面”。玩家应能在一个桌面里同时看到公屏记录、DM 状态、Agent 状态、线索牌堆、自己的剧本和手牌，而不是频繁切换 tab 查信息。

核心体验要求：

- 玩家明确知道当前阶段、阶段目标、DM 是否正在监听、哪个 Agent 准备回应、哪个 Agent 正在发言。
- 自动讨论默认开启，不需要玩家反复点击“让大家聊一轮”。
- DM 作为控场 Agent 主动监听公共讨论，并基于阶段目标自动推进。
- 线索以卡牌形式沉淀，可查看、可打出、可触发 Agent 讨论。
- 剧本、身份、目标、手牌应常驻可访问，不应脱离公屏。

### 15.2 桌面布局

```text
┌────────────────────────────────────────────────────────────────────────────┐
│ 顶栏：剧本名 / 当前阶段 / DM状态 / TTS状态 / 设置                         │
├───────────────┬───────────────────────────────────────┬────────────────────┤
│ 左侧玩家与状态 │ 中间公屏记录                            │ 右侧DM与线索区      │
│ 当前阶段       │ 公共讨论记录，字体更小                    │ DM主持台            │
│ 阶段目标       │ Agent发言随语音逐条沉淀                   │ 当前目标            │
│ 阶段完成度     │ DM只显示摘要                             │ 阶段判断            │
│ DM/Agent状态   │ 玩家发言保留高亮                           │ 协同监控            │
│ 角色头像状态   │ 等待态不进入历史                           │ 线索牌堆            │
├───────────────┴───────────────────────────────────────┴────────────────────┤
│ 底部玩家手牌区：身份卡 / 私密目标 / 我的线索 / 可用行动 / 剧本抽屉入口     │
└────────────────────────────────────────────────────────────────────────────┘
```

中间聊天区应缩小字体和气泡体积，作为“公屏记录纸”沉淀文字；语音仍是主要交付方式，但文字必须便于翻看历史。

### 15.3 DM 与 Agent 协同可视化

角色列表需展示轻量状态：

```text
DM       ● 监听中
沈明远   … 准备回应
周管家   ▶ 正在发言
陈默     ○ 等待
林晚     ? 等你回应
```

状态定义：

| 状态 | 含义 |
|------|------|
| ● 监听中 | DM 正在观察公共讨论 |
| … 准备中 | Agent 已被选中，正在生成回复 |
| ▶ 发言中 | 当前 Agent 正在 TTS 播放/公屏展示 |
| ✓ 已回应 | 本轮已经完成发言 |
| ○ 等待 | 当前无动作 |
| ? 等你回应 | 话头抛给玩家，需要玩家操作或发言 |

等待过程需要有临时可视化，例如“周管家正在组织语言...”或“DM 正在判断是否进入下一阶段...”。等待态不写入历史消息。

### 15.4 自动讨论机制

“让大家聊一聊”应改为默认开启的自动讨论模式：

- 进入自由交流、搜证、公开质询等开放阶段后，Agent 自动围绕当前议题讨论。
- 每轮 2-4 个 Agent 发言后，由 DM 判断继续讨论、等待玩家、发放线索或阶段收束。
- 玩家可暂停自动讨论，也可在合适时机插话。
- 若 TTS 播放队列仍在播报，玩家输入应避免与旧消息错位；系统可提示“角色发言中，稍候插话”。

### 15.5 DM 自动监听与阶段推进

DM 需要监听公共聊天区，并根据阶段目标推进流程。阶段推进不应主要依赖“应急推进”。

每个阶段需定义：

- 必须发生的事件：例如玩家发言、至少 N 位 Agent 回应、关键线索展示。
- 可选事件：例如私聊、额外质询、请求提示。
- 收束条件：DM 判断阶段目标达成后自动进入下一阶段。
- 等待玩家条件：需要玩家选择、展示线索、投票、提交判断时明确提示。

### 15.6 线索卡交互

线索卡应作为游戏对象，而不是普通文本：

- DM 发放的线索进入右侧下方线索牌堆。
- 玩家私有线索进入底部玩家手牌区。
- 线索卡可 hover 上浮、点击查看详情。
- 玩家可将线索“打出到公屏”，所有 Agent 可见并围绕它讨论。
- 线索可用于质询指定角色，形成“证据 + 问题 + 目标角色”的操作。

右侧线索牌堆：

```text
┌────────────────────┐
│ 线索牌堆            │
│   [卡1]             │
│     [卡2]           │
│       [卡3]         │
│ hover：上浮放大      │
│ click：打开详情      │
└────────────────────┘
```

### 15.7 多模态线索图

同一剧本的线索图必须风格一致、剧情连续，禁止按单张线索用互不相关的 prompt 独立生成。

线索图生成流程：

```text
剧本视觉圣经
  → 线索链路 storyboard
  → 一次性生成整组线索 contact sheet
  → 裁切成单张线索卡
  → 绑定到每张线索
```

线索图使用 Step Plan 图像能力，优先使用 `step-image-edit-2`。每个剧本应保存视觉风格、图像批次、线索顺序和线索间的剧情连接关系。

### 15.8 玩家操作面板

游戏不能只依赖语音或文本输入。每个阶段应暴露按钮式操作：

```text
可用行动
[展示线索] [质询角色] [选择搜证地点] [提交判断] [请求DM总结]
```

操作随阶段变化：

- 自由交流：展示线索、质询角色、请求总结。
- 搜证阶段：选择地点、查看线索、展示线索。
- 公开质询：选择角色、选择证据、发起追问。
- 投票阶段：投票、提交理由。
- 复盘阶段：查看真相、查看角色结局。

### 15.9 Step 模型配置要求

代码不得强制使用 `step-3.7-flash`。模型必须通过配置决定：

```env
STEP_BASE_URL=https://api.stepfun.com/step_plan/v1
STEP_MODEL=step-3.5-flash-2603
STEP_TTS_MODEL=stepaudio-2.5-tts
STEP_REALTIME_AUDIO_MODEL=stepaudio-2.5-realtime
STEP_ASR_MODEL=stepaudio-2.5-asr
STEP_IMAGE_MODEL=step-image-edit-2
```

当前推荐 `step-3.5-flash-2603` 作为默认 Agent 对话模型，原因是本项目存在高频、短轮次、多角色接话的调用形态，吞吐与响应稳定性优先于长推理深度；`step-3.7-flash` 可继续通过 `STEP_MODEL` 显式配置用于更重推理场景。

HTTP TTS/ASR 保留为稳定 fallback；Realtime 语音作为后续低延迟连续语音升级路径。

### 15.10 阅本阶段与随身剧本手册优化（v1.3）

当前实现校准：

- 阅本阶段虽然已经有 DM 开场与左侧应急推进，但中间主体验缺少明确的“阅本完成 → 开始游戏”入口，玩家不知道何时正式进入 DM 主持流程。
- 底部玩家手牌中的身份、目标等卡片都打开同一个完整剧本抽屉，导致“每张卡片内容一样”的重复感。
- 剧本抽屉以遮罩弹窗方式出现，会阻断玩家继续使用文字输入与语音输入，无法做到边读剧本边发言。

新增产品要求：

1. 阅本阶段主屏中必须出现明确的“游戏开始”按钮。
   - 玩家读完剧本后点击该按钮。
   - 点击后调用正常阶段推进，进入 DM 主持的下一阶段。
   - 左侧“应急推进”保留为 fallback，但主入口必须在中间游戏区。
2. 随身剧本手册必须是非阻塞阅读面板。
   - 打开剧本时不出现全屏遮罩。
   - 不阻断聊天输入框、语音输入按钮、TTS 播放队列。
   - 玩家可以一边看剧本，一边在公共频道发言或语音输入。
   - 自由讨论阶段中，Agent 正在发言或自动讨论请求未结束时，玩家输入框与麦克风仍必须可用；玩家可以先准备发言，但正式发送必须等待当前角色完整发言结束。
3. 玩家手牌卡片必须映射到不同剧本分区。
   - 身份卡：打开公开身份 / 私密背景。
   - 秘密卡：打开玩家秘密。
   - 目标卡：打开隐藏目标 / 胜利条件。
   - 案情卡：打开公共故事。
   - “剧本”按钮：打开完整目录或上次阅读分区。
4. 非阻塞手册必须支持分区切换。
   - 面板内部提供“全部 / 身份 / 秘密 / 目标 / 案情”等紧凑切换。
   - 切换分区只改变手册内容，不影响聊天区状态。

### 15.11 玩家发言队列与去模板化角色发言（v1.4）

新增体验目标：

1. 玩家发言必须进入“待发送队列”。
   - 玩家可随时通过文字输入或按住说话准备发言。
   - ASR 先把语音转换为文字，再复用同一套待发送/正式发送流程；如果已有文字草稿，则与语音转写合并为一条玩家发言。
   - 玩家确认发送后，如当前有角色/DM 正在发言或 TTS 队列未空，该消息不立即进入公屏。
   - 等待期间展示“转圈图标 + 等待上一位发言结束…”，让玩家明确知道消息已排队。
   - 当前角色完整播放结束后，玩家消息才正式插入公屏并发送给后端。
2. 角色语音不得被玩家输入打断。
   - 玩家开始输入、按住说话、点击发送，均不得停止当前 TTS。
   - 同一角色的一条长发言必须完整朗读完毕，再轮到下一位角色。
   - 长文本应分段连续朗读，而不是截断后直接进入下一条。
3. 真实模型模式不得混入固定 mock 角色台词。
   - 配置真实模型 key 时，角色发言失败不得 fallback 到固定 mock 句式。
   - 失败时应显示轻量等待/重试状态，或记录错误供调试，不伪装成角色正常发言。
   - 无 key 的离线 mock 只能作为明确标记的演示模式。
4. 角色发言需要降低 AI 味和重复性。
   - 每个角色必须有专属语言风格卡，包括语气、句式、反问倾向、回避方式、情绪强度。
   - Prompt 不得过度模板化，不应反复要求固定“2-4句/反驳/点名/递话头”结构。
   - 需要对最近话题做去重，避免围绕同一角色和同一证据反复打转。

### 15.12 线索卡共享与多模态推进（v1.5）

#### 15.12.1 当前实现缺口（第一遍校准）

当前线索卡已经具备基础 UI，但还不是真正的剧本杀证据对象：

1. 线索卡没有稳定媒体资产。
   - 当前 `ClueDeck` 会用前端 SVG 占位图兜底，但剧本数据本身没有 `imageUrl`、`mediaType`、`styleGuide` 或 `visualBatchId`。
   - 内置剧本和 AI 生成剧本都只保存文字线索。
2. “打出到公屏”缺少事件语义。
   - 玩家打出线索时，本质是发送一条普通玩家消息，内容为“【展示线索】标题 + 内容”。
   - 系统没有记录展示者、展示目标、展示范围、展示动作类型和相关线索 ID 的结构化含义。
3. Agent 对线索的感知不稳定。
   - 角色 Agent 只能从普通聊天历史中读到线索文本。
   - `conversation-director` 无法把“某张线索卡被展示”作为高优先级事件，也无法确保被点名/被证据指向的角色先回应。
4. Agent 之间没有互相展示线索的能力。
   - 当前只有玩家可以触发线索展示。
   - AI 角色不能主动拿出线索质询别人、补充证据或把某张公共线索递给另一位角色。
5. DM 多模态推进不足。
   - DM 发放线索时已有 TTS 播报，但缺少同步展示图片/视频的主持动作。
   - 视频生成能力未作为稳定依赖确认，因此 V1.5 只落地图片，视频保留可选媒体位。

#### 15.12.2 目标体验（第二遍目标方案）

V1.5 的目标是让线索卡成为所有角色共享的桌面证据，而不是附在聊天里的文字：

1. 每个剧本必须有一组风格一致的线索图片卡。
   - 内置测试剧本需要预置图片卡资产。
   - AI 生成剧本需要输出线索视觉规划，再通过 Step 图像模型预生成整组图片。
   - 同一剧本的线索图必须共享视觉圣经：时代、地点、色调、光线、构图、物件材质和反复出现的视觉符号。
2. 线索图片必须有剧情连续性。
   - 不能为每张线索单独写无关 prompt。
   - 必须先生成“线索链路 storyboard”，再按同一视觉风格生成整组图片。
   - 图片应支持玩家快速理解“这张卡为什么重要”，而不是只做氛围装饰。
3. 玩家可以把线索展示给所有人或指定角色。
   - 展示给所有人：公屏出现线索卡消息，所有 Agent 可见并围绕该卡讨论。
   - 指定角色质询：玩家选择线索 + 目标角色 + 问题，目标角色必须优先回应。
   - 私下展示：后续预留给私聊阶段，不作为本轮必做项。
4. AI 角色也可以互相展示线索。
   - 在自由交流、搜证、公开质询阶段，Agent 可基于自己的立场主动展示已公开线索。
   - Agent 展示线索时必须有明确动机：自证、反驳、转移怀疑、逼问他人或补充时间线。
   - 被线索指向或被点名的角色必须优先接话。
5. DM 发放线索必须同步多模态展示。
   - 进入线索阶段时，DM 先用 TTS 说明阶段目标和线索背景。
   - 右侧线索区同步亮起/翻开对应图片卡。
   - 公屏只沉淀短摘要，例如“DM 发放线索《死亡后指令》”。
   - 详细线索内容、图片和历史发放记录保留在右侧线索区。
6. 视频作为可选媒体位，不阻塞 V1.5。
   - 若后续确认 Step 或其他已配置服务有稳定视频生成 API，可为线索卡补 `videoUrl`。
   - 当前实现先支持 `mediaType=image|video|none`，但只生成和展示图片。

#### 15.12.3 测试剧本资产要求

本轮开发需要至少选择 2 个内置测试剧本补齐线索图资产：

1. 《零号舱的悖论》

### 15.13 阶段举证、共识收敛与 DM 主导推进（v1.6）

#### 15.13.1 当前实现缺口（第一遍校准）

当前游戏阶段仍然过于像“开放式群聊”：

1. 阶段缺少明确的议题结构。
   - 每个阶段有名称和描述，但没有明确声明“本阶段必须完成哪些推理/情感/时间线任务”。
   - DM 面板有完成度，但更多是“是否发言/是否释放线索”的状态，而不是“是否解决了本阶段问题”。
2. 举证环节没有成为阶段核心动作。
   - 搜证、质询、推理等阶段理论上应围绕线索卡展开，但实际体验仍是公聊气泡为主。
   - 玩家和 Agent 缺少“拿这张线索证明/反驳/质询某人”的显式操作。
3. 阶段缺少共识/分歧收敛。
   - 一轮讨论后，系统没有主动整理“大家目前同意什么、不同意什么、谁还在反对、下一步该核查什么”。
   - 当无法形成共识时，DM 也没有明确将“未达成共识”作为一个可推进状态处理。
4. 玩家缺少阶段结论输入。
   - 玩家只能继续聊天或应急推进，不能提交“我认为本阶段结论是……”。
   - 这让 DM 无法基于玩家的真实判断来决定推进、补线索或继续质询。

#### 15.13.2 目标体验（第二遍目标方案）

每个可讨论阶段都应形成一个闭环，而不是无限聊天：

```text
DM 开阶段议题
  → DM / 玩家 / Agent 举证
  → 围绕证据讨论和质询
  → DM 做共识检查
  → 玩家确认阶段结论，或标记无法达成共识
  → DM 收束并决定推进 / 补线索 / 继续质询
```

阶段推进的体验规则：

1. DM 必须给出本阶段议题。
   - 例如：“本阶段要确认案发前后 30 分钟内每个人的位置。”
   - 议题显示在右侧 DM 主持台，不刷屏占据公屏。
2. 举证是阶段动作，不是普通聊天。
   - 玩家可选择线索卡并执行“展示给大家”“质询角色”“支持我的推理”“反驳某人说法”。
   - Agent 可主动引用已公开线索来支持或反驳。
   - DM 可提示“目前缺少证据支撑，请先选择一张线索卡”。
3. 共识板需要持续更新。
   - “已达成共识”：例如“罗维两点后指令不可信”。
   - “仍有分歧”：例如“苏砚药柜记录是否被盗用”。
   - “待核查”：例如“秦砚是否拥有日志维护权限”。
4. 无法达成共识也是有效状态。
   - 当角色意见冲突明显且短期无法统一，玩家可以点“标记无法达成共识”。
   - DM 应把分歧记录下来，并决定是补发线索、要求指定角色回应，还是带着分歧进入下一阶段。
5. 玩家需要能提交阶段结论。
   - 玩家可点击“提交阶段结论”，用一句话说明当前判断。
   - DM 根据结论判断：方向正确则推进；缺证据则要求举证；偏离过远则给轻提示或安排质询。
6. DM 收束必须基于结构化状态。
   - 不能只因为“聊了几句”就推进。
   - 至少满足其一：关键线索已展示并讨论、玩家提交阶段结论、共识/分歧已明确、DM 判断继续讨论收益低。

#### 15.13.3 玩家可用操作

阶段底部动作区应按阶段显示：

```text
[展示线索] [用线索质询] [请求共识检查] [提交阶段结论] [标记无法达成共识] [请求 DM 收束]
```

动作说明：

- 展示线索：把线索卡公开到公屏，所有 Agent 必须可见。
- 用线索质询：选择线索 + 目标角色 + 问题，目标角色优先回应。
- 请求共识检查：DM 总结当前“共识/分歧/待核查”。
- 提交阶段结论：玩家输入当前判断，由 DM 判定是否足够推进。
- 标记无法达成共识：玩家认为当前信息不足或角色僵持，要求 DM 记录分歧。
- 请求 DM 收束：不是强制跳阶段，而是让 DM 根据阶段状态决定推进或补充行动。

#### 15.13.4 DM 与 Agent 行为要求

- DM 必须持续监听公共讨论导向，避免角色只是在各说各话。
- 当讨论偏离阶段议题时，DM 应用短提示拉回议题。
- 当某位 Agent 的观点被线索直接反驳时，该 Agent 必须回应。
- 当多位角色形成一致判断时，DM 应标记“初步共识”，并询问玩家是否认可。
- 当分歧长期存在时，DM 应明确说出“目前无法达成共识的点”，而不是继续让大家重复讨论。
- 阶段收束时，DM 应输出“本阶段结论摘要 + 遗留问题 + 下一阶段目标”，其中公屏只保留摘要，详情进入右侧主持台。
